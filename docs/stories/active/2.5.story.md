# Story 2.5: Execucao de Limpeza com Progresso e Logging

## Metadata
- **Epic:** 2 - System Cache Cleaner
- **Status:** Draft

- **Depends On:** 2.1, 2.2, 2.3, 2.4

## User Story
Como usuario do Windows Optimizer, eu quero que a limpeza de cache execute de forma confiavel com feedback de progresso em tempo real, para que eu saiba o que esta sendo deletado e tenha confirmacao do espaco liberado.

## Description
Implementar o metodo `clean()` do `CacheCleaner` com emissao de eventos de progresso via Tauri. O frontend ja exibe o `ProgressOverlay` (Story 2.4) — esta story conecta o backend. Implementar a logica real de delecao usando `platform/filesystem.rs` (Story 2.1), com logging de cada operacao (NFR-006) e tratamento correto de arquivos em uso.

## Acceptance Criteria
- [ ] AC-1: Tauri command `clean_system_cache` aceita `{ categories: Vec<String> }` e executa limpeza das categorias selecionadas
- [ ] AC-2: Durante execucao, emite eventos Tauri `"cleanup-progress"` com payload `{ step: String, percent: u8, current_category: String }`
- [ ] AC-3: Eventos de progresso sao emitidos para cada categoria (ex: 0%, 20%, 40%, 60%, 80%, 100%)
- [ ] AC-4: Arquivos em uso (locked) sao ignorados silenciosamente — contados em `CleanResult.files_skipped`, logados mas nao causam falha
- [ ] AC-5: Limpeza de Windows Temp verifica elevacao antes — retorna `AppError::ElevationRequired` se sem admin
- [ ] AC-6: Limpeza de Prefetch verifica elevacao antes — retorna `AppError::ElevationRequired` se sem admin
- [ ] AC-7: Limpeza de Windows Update cache verifica servico wuauserv antes — retorna aviso se servico rodando
- [ ] AC-8: Cada arquivo deletado gera entrada de log: `[timestamp] [cache_cleaner] [delete] [OK/SKIP] {path} {bytes}`
- [ ] AC-9: `CleanResult` final inclui: `total_freed_bytes`, `files_deleted`, `files_skipped`, `warnings`, `errors`
- [ ] AC-10: Esvaziar Recycle Bin usa `SHEmptyRecycleBin` API (Story 2.3) — nao delecao manual

## Technical Notes

### Fluxo de dados (arquitetura secao 1.2)
```
Frontend: invoke("clean_system_cache", { categories })
    -> commands::cache::clean_system_cache()
        -> modules::cache_cleaner::CacheCleaner::clean()
            -> emite eventos "cleanup-progress"
            -> platform::filesystem::safe_delete_dir_contents()
        -> retorna CleanResult
Frontend: listen("cleanup-progress") -> atualiza ProgressOverlay
Frontend: recebe CleanResult -> exibe resumo
```

### Emisao de eventos em modules/cache_cleaner.rs
```rust
use tauri::AppHandle;

pub async fn clean(
    &self,
    app: &AppHandle,
    categories: Vec<String>,
) -> Result<CleanResult, AppError> {
    let total = categories.len();
    for (i, cat_id) in categories.iter().enumerate() {
        let percent = ((i as f32 / total as f32) * 100.0) as u8;
        app.emit("cleanup-progress", CleanupProgress {
            step: format!("Limpando {}", cat_id),
            percent,
            current_category: cat_id.clone(),
        }).ok(); // nao falhar se frontend desconectado

        match cat_id.as_str() {
            "user_temp" => self.clean_user_temp().await?,
            "windows_temp" => self.clean_windows_temp().await?,
            "chrome" => self.clean_browser_cache("chrome").await?,
            // ...
        }
    }
    app.emit("cleanup-progress", CleanupProgress {
        step: "Concluido".into(),
        percent: 100,
        current_category: "".into(),
    }).ok();
    Ok(self.build_clean_result())
}
```

### CleanupProgress struct
```rust
#[derive(Debug, Clone, serde::Serialize)]
pub struct CleanupProgress {
    pub step: String,
    pub percent: u8,
    pub current_category: String,
}
```

### Frontend: listen ao evento
Em `CacheCleaner.tsx`, usar `useEffect` com `listen("cleanup-progress", ...)` do `@tauri-apps/api/event` para atualizar o `ProgressOverlay`.

### Logging por arquivo (NFR-006)
```rust
tracing::info!(
    module = "cache_cleaner",
    action = "delete",
    result = "OK",
    path = %file_path.display(),
    bytes = file_size,
    "File deleted"
);
```

### Tratamento de erros atomicos
A limpeza e "best effort" — falhas individuais de arquivo nao abordam a operacao inteira. Apenas `AppError::ElevationRequired` causa retorno antecipado.
